项目背景

AI产品故事

🚀 公司愿景

我在 星球家园（深圳）科技有限公司 实习期间，公司致力于打造 Agent 平台与生态——让各类 App 能把自己的 API 封装为"可被 Agent 调用的工具"，沉淀一个跨应用的服务市场。

👥 团队构成

团队共4人：我（AI产品经理）+ 老板（对接/BD）+ 算法同学（LLM/RAG）+ 前后端同学。我负责从0到1的需求澄清与范围锚定、PRD、关键链路设计、RAG结构、工具契约、原型说明、指标与验收。

🎯 MVP切口选择

项目名称：货拉拉下单Agent意图识别

Agent平台

需求描述：老板与货拉拉沟通过后，决定先做一个 "货拉拉下单 Agent" 作为平台的第一个行业样板，用来验证：

验证目标：
  • 意图识别稳定性：Agent 对自然语言意图识别与参数抽取的稳定性
  • 工具契约联动：工具契约/供给联动能否把"开放世界任务"拉回到可控的自动化链路
  • RAG知识库效果：RAG 知识库能否有效降低车型选择、装载理解带来的决策摩擦
  • 平台化能力：为后续多App API生态接入打基础

核心目标：让用户一句话就下单，就像客服一样能问两句把信息补齐，能推荐车型，给个价，最好还能马上下单。

💬 多轮需求获取纪要

1

会议#1 Kickoff（第1周）

参与：老板、我、算法、FE/BE

  • 城市范围：先做深圳；覆盖"搬家/大件运输"
  • 核心体验：语音优先、按钮优先（尽量不打字）；车型推荐必须解释
  • 风险与护栏：不做支付/派单；草稿阶段明确"非真实下单"
  • KPI指标：抽取准确率≥85%，澄清≤2轮，草稿一次通过≥70%，P95≤10s

行动项：我拉出MVP功能清单+流程图；算法准备"货物→车型"的RAG结构样例；FE/BE梳理工具契约

2

会议#2 深挖与对齐（第2周）

争议点：API直连（真实报价/供给）还是先做模拟供给？

  • 结论：MVP阶段供给API可用"内部模拟层"（可插拔），接口契约先定死，未来接真系统
  • 新增功能：用户长时间未选车型（20s）触发再次引导；RAG必须可解释（给装载示例）
  • 行动项：我加入"沉默超时"的分支与UI文案；算法整理"按摩椅/书箱/行李"等货物模板

关键决策：供给API可用"内部模拟层"，便于未来切真实系统

3

会议#3 方案评审（第3周）

确认方案：

  • 车型推荐：三选按钮（🚚 推荐/🚛 更大/🚐 更小）+供给角标（高/中/低）
  • 地址处理：地址模糊→候选三选，不强迫用户手打
  • 交互指标：按钮/语音交互占比≥80%
  • 行动项：我完善PRD、原型说明&埋点；算法few-shot样例与正负例收集

最终确定：冻结MVP范围：仅"到草稿确认"；不含定价最终口径、不含支付

⚙️ 关键设计与技术权衡

🎯

下单闭环

先到草稿（不触达支付/派单），风险可控、上线快；把"可观测&复盘"跑通

🧠

NLP处理

LLM + 结构化抽取（schema/正则/正例），兼顾泛化与可控，易做A/B与回放

⏰

时间解析

模糊→标准（rules+LLM兜底），确保"明天上午/11点"等口语解析稳定

📍

地址解析

地图API+候选Top3，降低手打概率，缩短时延

🚚

车型推荐

三选（推荐/更大/更小）+ RAG可解释，降认知负担；避免用户"选型焦虑"

🔗

供给联动

统一`/supply/check`，MVP可模拟，便于未来切真实系统；前端角标"高/中/低"

泳道图｜Agent意图识别主链路

技术架构

从语音输入→意图识别→参数抽取→车型推荐→草稿确认；以 RAG知识库 和 工具契约 为核心，确保抽取准确性和推荐合理性。

        sequenceDiagram
        autonumber
        participant U as 用户
        participant A as Agent（界面&编排）
        participant N as 意图与抽取
        participant M as 地图API（地址解析）
        participant T as 时间API（模糊→标准）
        participant K as RAG知识库（车型×可装物品）
        participant S as 供给API（时段/区域供给）
        participant C as 校验器（完整性/一致性/约束）
        participant O as 输出（订单草稿）
    
        %% 顶部引导与示例
        A-->>U: 顶部提示：您可以直接说出需求（运什么、从哪到哪、何时）\n示例：明天上午11点从罗湖口岸XX小区搬到南山区置地大厦D栋，按摩椅、四大项行李
        Note over U,A: 原则：尽量用按钮/语音澄清，避免用户打字
    
        %% 用户输入 & 规划解析顺序
        U->>A: 语音/文本输入（如上示例）
        A->>N: 解析意图与抽取字段
        N-->>A: intent=下单, time=2025-09-05 11:00, from=罗湖口岸XX小区, to=南山区置地大厦D栋, cargo=按摩椅+4件行李, vehicle=未指定
    
        %% 时间解析（优先级1）
        A->>T: 校准时间
        T-->>A: 返回标准时间 2025-09-05 11:00
    
        %% 地点解析（优先级2）
        A->>M: 地址标准化（from/to）
        M-->>A: 标准地址与坐标（罗湖口岸XX小区 → 南山区置地大厦D栋）
    
        %% 货物与车型推荐（优先级3/4）
        A->>K: 检索“按摩椅+大件行李”装载范例
        K-->>A: 推荐：中面/小货车；按摩椅需空间，4件行李可放小货车
        A->>S: 查询 2025-09-05 11:00 该区域车型供给
        S-->>A: 可用：小货车(高)，中面(中)，厢货(低)
    
        %% 三选推荐 - 可视化按钮（附车图标）
        A-->>U: 根据您的需求推荐 🚚 **小货车**（可装行李+中型家具）\n也可选 🚛 **更大车型** 或 🚐 **更小车型**\n按钮：[🚚 推荐小货车｜🚛 更大车型｜🚐 更小车型]
        opt 用户长时间未选择
            A-->>U: 提醒：按摩椅体积较大，是否考虑更大车型？\n按钮：[保持推荐｜更大｜更小]
        end
    
        %% 用户选择车型
        alt 用户点“推荐小货车”
            U-->>A: 选择 推荐小货车
            A->>K: 获取小货车装载范例
            K-->>A: 返回装载清单与注意事项
        else 用户点“更大车型”
            U-->>A: 选择 更大车型
            A->>K: 获取中面/厢货装载范例
            K-->>A: 返回装载清单与注意事项
        else 用户点“更小车型”
            U-->>A: 选择 更小车型
            A->>K: 获取面包车/小面范例
            K-->>A: 返回装载清单与注意事项
        end
    
        %% 完整性校验
        A->>C: 校验字段完整性与约束
        alt 缺失/冲突
            C-->>A: 缺少字段：是否需要搬运上楼？
            A-->>U: 澄清按钮：[需要搬运｜无需搬运]
            U-->>A: 选择“需要搬运”
            A->>C: 重新校验
            C-->>A: 通过
        else 完整
            C-->>A: 全部通过
        end
    
        %% 生成草稿
        A->>O: 生成订单草稿（结构化数据）
        O-->>U: 展示草稿（含时间/地址/车型/注意事项）\n支持一键修改或语音修改
    
        %% 用户确认/修改
        opt 用户修改
            U-->>A: 修改时间/车型
            A->>T: 校准时间
            T-->>A: 返回新时间
            A->>S: 查新供给
            S-->>A: 返回结果
            A->>C: 校验
            C-->>A: 通过
            A->>O: 更新草稿
            O-->>U: 展示更新草稿
        end
    
        alt 用户确认草稿
            U-->>O: 确认草稿
            O-->>A: 返回最终订单数据
            A-->>U: 草稿确认完成，可进入下单环节（MVP不含支付/派单）
        else 用户取消
            U-->>A: 取消
            A-->>U: 草稿已保存，随时可继续
        end
